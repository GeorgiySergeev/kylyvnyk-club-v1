ROLE: Senior backend engineer focused on security and observability.

CONTEXT:
- Block 5 of 35. @kyc/api router skeleton is in place.
- We need two cross-cutting concerns wired into tRPC before we add real procedures:
  1. Rate limiting on public/auth-sensitive endpoints (applications.create, auth.register, auth.requestMagicLink, push.subscribe, reviews.create, chat.sendMessage).
  2. Audit logging for every admin mutation (creates, updates, deletes).

GOAL:
Add a rate limiter using Upstash Redis (REST API) and an audit log middleware to @kyc/api, wire both into the procedure helpers, and document configuration.

REQUIREMENTS:

1. Install: `@upstash/redis`, `@upstash/ratelimit`.

2. `packages/api/src/lib/redis.ts`:
   - Lazy singleton Redis client using `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN`.
   - Graceful fallback in tests/dev: if env vars missing, return an in-memory shim implementing get/set/eval enough for @upstash/ratelimit (use `Ratelimit.cachedFixedWindow` or wrap an in-memory limiter).

3. `packages/api/src/lib/ratelimit.ts`:
   - Export factory `createLimiter(name: string, opts: { tokens: number; window: '1 s' | '10 s' | '1 m' | '1 h' })`.
   - Pre-built limiters: 
     - `applicationsLimiter` (5 per hour per IP)
     - `authLimiter` (10 per 10 minutes per IP)
     - `chatLimiter` (30 per minute per user)
     - `reviewLimiter` (5 per hour per user)
     - `genericPublicLimiter` (60 per minute per IP)
   - Key strategy: `userId ?? ip ?? 'anon'`.

4. `packages/api/src/middleware/rateLimit.ts`:
   - tRPC middleware factory `withRateLimit(limiter)`.
   - On limit exceeded: throw TRPCError code TOO_MANY_REQUESTS with `cause: { retryAfterMs }`.
   - Headers exposed via `ctx.res?.setHeader` if the runtime supports it (optional).

5. `packages/api/src/middleware/audit.ts`:
   - tRPC middleware `withAudit({ action, entity, captureBefore?, captureAfter? })`.
   - After successful procedure execution, insert an AuditLog row with: actorId (from session), action, entity, entityId (extracted from input/output via passed selectors), beforeJson, afterJson, ip, userAgent.
   - Failures in audit write must NOT fail the parent request — wrap in try/catch and log to console.

6. Update `packages/api/src/trpc.ts`:
   - `publicProcedure` stays as-is.
   - Add `publicLimitedProcedure(limiter)` — returns a procedure builder with rateLimit applied.
   - `memberProcedure` and `adminProcedure` accept optional `{ rateLimit?: RateLimiter; audit?: AuditOptions }` via chained `.use()` helpers — document the pattern in README.
   - Add helper `auditedAdminProcedure(opts)` that bundles rate-limit + audit + role check.

7. Update at least 4 existing stubs in routers to apply the new middlewares (still NOT_IMPLEMENTED returns, but with middlewares chained — proves wiring works):
   - `applications.create` → applicationsLimiter
   - `auth.register` and `auth.requestMagicLink` → authLimiter
   - `admin.partners.create` → auditedAdminProcedure({ action: 'partner.create', entity: 'Partner' })

8. `packages/api/.env.example` documenting UPSTASH_REDIS_REST_URL, UPSTASH_REDIS_REST_TOKEN.

9. Unit tests with Vitest in `packages/api/src/__tests__/`:
   - rateLimit: simulate 6 calls with limit 5 → 6th throws TOO_MANY_REQUESTS.
   - audit: stub Prisma, verify AuditLog.create called with correct payload, verify primary procedure success when audit write throws.

10. Add `test` script to `packages/api/package.json` running vitest. Add `vitest` + `@vitest/coverage-v8` as devDeps.

ACCEPTANCE CRITERIA:
- `pnpm --filter @kyc/api test` passes.
- `pnpm --filter @kyc/api typecheck` passes.
- In dev without Upstash env vars, the app still runs (in-memory fallback).
- README section explains how to wire the middlewares to new procedures.

DON'Ts:
- Do not couple rateLimit to a specific transport (must work for tRPC in both Edge and Node runtimes).
- Do not block on audit log writes — fire-and-forget with error logging only.
