ROLE: Senior fullstack engineer. Build the type-safe API layer.

CONTEXT:
- Block 4 of 35. @kyc/db is ready with the Prisma schema and a singleton client.
- We use tRPC v11. The API package exports the AppRouter type; both apps/web and apps/admin will mount it via Next.js Route Handlers in later blocks.

GOAL:
Create `@kyc/api` with a tRPC v11 setup, Zod schemas, procedure helpers (public/member/admin), and stubbed routers for every domain. No business logic yet — just typed signatures and "TODO" returns where appropriate. Subsequent blocks will fill in implementations.

REQUIREMENTS:

1. `packages/api/package.json`:
   - name `@kyc/api`, type "module"
   - deps: @trpc/server@^11, zod@^3, superjson, @kyc/db (workspace:*)
   - peerDeps: none
   - exports: `.` (root index), `./client` (types only), `./schemas`

2. `packages/api/src/trpc.ts`:
   - Initialize tRPC with superjson transformer.
   - Create `Context` type: `{ session: Session | null; prisma: PrismaClient; ip: string | null; userAgent: string | null }`.
   - `createContext()` factory — signature only; concrete implementation provided by each Next.js app.
   - Helpers: `publicProcedure`, `memberProcedure` (throws UNAUTHORIZED if no session or role !== MEMBER+), `adminProcedure` (role ADMIN or SUPER_ADMIN), `superAdminProcedure`.
   - Error formatter that exposes Zod issues.

3. `packages/api/src/schemas/` — Zod schemas grouped by domain. One file per domain (auth, member, partners, categories, geo, favorites, applications, deals, reviews, referrals, events, promotions, chat, push, onboarding, admin). Examples:
   - `partners.ts`: PartnerListInput (cursor pagination, filters: countryId?, cityId?, categoryIds?[], search?, sort?), PartnerListItem, PartnerDetail, PartnerCreateInput (admin), PartnerUpdateInput.
   - `applications.ts`: ApplicationCreateInput { name min 2, email .email(), phone min 5, countryId cuid, source enum, message max 1000, consent literal(true) }.
   - All schemas exported as both Zod object AND `z.infer` types.

4. `packages/api/src/routers/` — one file per domain router, each with stubs:
   - `auth.ts`: register, requestMagicLink, verifyMagicLink, me, updateProfile, enable2FA, verify2FA, disable2FA, listSessions, revokeSession.
   - `member.ts`: getDashboard, getCardData, getOnboardingProgress, completeOnboardingStep.
   - `partners.ts`: list (public, cursor pagination), bySlug, related.
   - `categories.ts`: list.
   - `geo.ts`: listCountries, listCities (by countryId).
   - `favorites.ts`: list, toggle.
   - `applications.ts`: create (public).
   - `deals.ts`: list, get.
   - `reviews.ts`: create, listByPartner.
   - `referrals.ts`: getMyCode, stats, list.
   - `events.ts`: list, bySlug, register, cancelRegistration, myEvents.
   - `promotions.ts`: listActive, byPartner.
   - `chat.ts`: getOrCreateConversation, listMessages (cursor), sendMessage, markRead.
   - `push.ts`: subscribe, unsubscribe.
   - `onboarding.ts`: listSteps.
   - `admin.ts` — sub-router with: partners, categories, geo, applications, members, events, promotions, chat (inbox), push (campaigns), reviews (moderation), onboarding (editor), audit, users, dashboard. Each is its own router with full CRUD stubs.
   - Each procedure currently returns `throw new TRPCError({ code: 'NOT_IMPLEMENTED', message: 'TODO: block N' })` where N is the block that will implement it (reference the plan).

5. `packages/api/src/root.ts`:
   - Compose all routers into `appRouter`.
   - Export `type AppRouter = typeof appRouter` and the router instance.

6. `packages/api/src/index.ts`: re-export `appRouter`, `AppRouter`, `createContext`, schemas, and procedure helpers.

7. `packages/api/src/client.ts`: type-only re-exports for consumers (apps).

8. README.md describing: how to add a new router, the procedure helper hierarchy, error code conventions, the "stub returns" convention.

ACCEPTANCE CRITERIA:
- `pnpm --filter @kyc/api typecheck` passes (strict TS).
- `import { appRouter, type AppRouter } from '@kyc/api'` works from a downstream package.
- Zod schemas exported and discoverable via `import { partnerListInput } from '@kyc/api/schemas'`.
- All 16 domain routers exist with at least one procedure each.

DON'Ts:
- Do not implement any business logic — every procedure throws NOT_IMPLEMENTED.
- Do not depend on Next.js, React, or any browser globals.
- Do not import `@kyc/db` in schemas — only in routers.
- Do not create the HTTP handler — that belongs to each app.
