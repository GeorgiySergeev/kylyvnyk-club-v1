# ТЗ / организованный план

## Роадмап: «Как стать AI Engineer в 2026»

## 1. Назначение документа

Этот документ переводит исходный роадмап в формат понятного плана работ и технического задания на самостоятельное обучение и сборку практических AI-agent проектов.

Документ предназначен для:

- самостоятельного изучения профессии AI Engineer;
- трекинга прогресса по фазам;
- использования как основы для личного учебного плана;
- адаптации под командное обучение, менторство или внутреннюю академию.

## 2. Цель

За 17 недель пройти путь от базового понимания agentic AI до сборки production-ready агентной системы с:

- инструментами;
- памятью;
- durable execution;
- middleware;
- системой оценок;
- regression harness;
- production hardening.

## 3. Ожидаемый результат

По завершении плана исполнитель должен уметь:

- отличать workflow от agent-подхода;
- проектировать цикл агента и tool dispatch;
- строить stateful и persistent агентов;
- работать с LangGraph, Claude Agent SDK и смежной обвязкой;
- организовывать контекст-инжиниринг по схеме Write / Select / Compress / Isolate;
- реализовывать память, хуки, sandboxing и observability;
- собирать evals, golden datasets и CI-gates;
- доводить агента до production-ready состояния.

## 4. Формат реализации

- Длительность: 17 недель.
- Формат: 6 фаз.
- Режим: 10–15 часов в неделю.
- Результат каждой фазы: не теория, а конкретный артефакт — код, проект, документ, eval-suite или production-конфигурация.
- Подход: «сначала собрать, потом оптимизировать, затем измерять».

## 5. Основные принципы

- Не изучать сразу много фреймворков.
- Делать упор на обвязку (harness), а не только на модель.
- Изучать примитивы, которые переносятся между инструментами.
- Каждую фазу завершать проверяемым результатом.
- Любое улучшение подтверждать трейсам, метриками или evals.

## 6. Основные сущности обучения

### 6.1. Ключевые концепции

- Augmented LLM.
- Workflow vs Agent.
- Harness Engineering.
- Context Engineering.
- Tool Use.
- Middleware.
- Sub-agents.
- Memory.
- Durable Execution.
- Observability.
- Evals.
- Production Hardening.

### 6.2. Четыре примитива контекст-инжиниринга

- **Write** — запись промежуточных данных, scratchpads, memory-файлы.
- **Select** — выбор релевантного контекста в момент использования.
- **Compress** — сжатие истории или результатов при переполнении окна контекста.
- **Isolate** — выделение отдельных окон контекста для суб-агентов.

## 8. Детализация фаз

## 8.1. Фаза 0. Основы

### Цель

Понять базовые конструкции agentic AI до написания серьезного кода.

### Что изучить

1. Разницу между workflow и agent.
2. Концепцию augmented LLM.
3. Context engineering как отдельную дисциплину.
4. Обвязку как операционную систему агента.
5. Текущее состояние agent engineering в 2026.

### Практика

Подготовить личный документ, в котором своими словами объяснить:

- workflow vs agent;
- augmented LLM;
- Write / Select / Compress / Isolate;
- orchestrator-worker;
- разницу между harness, model и framework;
- ожидаемые режимы сбоев.

### Критерий завершения

Исполнитель умеет объяснить основы без опоры на фреймворк-специфичный жаргон.

## 8.2. Фаза 1. Первый простой агент

### Цель

Понять, как работает tool-using агент на низком уровне и что дает готовая обвязка.

### Что изучить

1. Ручной цикл агента: model -> tools -> model.
2. stop_reason, tool_use, tool_result, parallel tool calls.
3. Claude Agent SDK как reference harness.
4. Skills, hooks, sub-agents, CLAUDE.md.

### Практика

#### Проект 1

Собрать простого агента без фреймворка примерно в 100 строк:

- web_search;
- read_file;
- write_file.

#### Проект 2

Переписать того же агента на Claude Agent SDK:

- добавить `CLAUDE.md`;
- добавить 1 skill (`SKILL.md`);
- добавить 1 post-tool hook;
- заспавнить 1 sub-agent.

#### Мини-отгрузка

Сделать полезный утилитарный агент, который:

- запускается по расписанию;
- читает локальные markdown-заметки и RSS;
- пишет утренний briefing на диск.

### Критерий завершения

Исполнитель умеет:

- написать свой агентный цикл вручную;
- объяснить, что именно автоматизирует SDK;
- поддерживать маленький прикладной агент, который реально используется.

## 8.3. Фаза 2. Настоящий агент с нормальной архитектурой

### Цель

Построить многошагового persistent/stateful агента на LangGraph + Deep Agents.

### Базовый стек

- LangGraph 1.0;
- LangChain `create_agent`;
- Deep Agents;
- PostgresSaver;
- LangSmith;
- Tavily или Firecrawl;
- при необходимости MCP / Composio / Arcade.

### Что изучить

1. Runtime LangGraph: state graph, nodes, edges, checkpointers.
2. Middleware как основной слой кастомизации.
3. Tools, MCP и code execution pattern.
4. Память без преждевременного ухода в vector DB.

### Практический проект

Собрать deep-agent «research analyst»:

- lead-agent принимает исследовательский вопрос;
- строит план;
- пишет TODO в виртуальную ФС;
- спавнит 3 поисковых sub-agents параллельно;
- суб-агенты пишут результаты в файлы;
- citation-agent проверяет утверждения;
- writer-agent собирает финальный markdown-отчет;
- все состояние сохраняется через `PostgresSaver`;
- агент умеет продолжать работу после kill процесса;
- есть HITL-подтверждение при превышении токен-бюджета;
- весь pipeline запускается через `make demo`.

### Критерий завершения

Исполнитель умеет собирать stateful-агента с persistence, interrupts, middleware и трассировкой.

## 8.4. Фаза 3. Собственная обвязка

### Цель

Понять устройство harness engineering через реализацию собственного тонкого слоя обвязки.

### Компоненты, которые нужно реализовать

- Loop control.
- Tool registry и dispatch.
- Context management.
- Persistence.
- Sub-agent orchestration.
- Skills и progressive disclosure.
- Hooks.
- Observability.
- Sandboxing.
- Secret/auth brokering.

### Практический проект

Собрать mini-harness на Python (~1500 строк), включающий:

- цикл вокруг `anthropic.messages.create` или LiteLLM;
- декоратор `@tool` с JSON-schema;
- загрузчик системных правил из `./harness/rules/*.md`;
- загрузчик `SKILL.md` с прогрессивной загрузкой;
- примитив sub-agent spawn с isolated context;
- offload длинных tool results в файловую систему;
- авто-компакцию при 85% окна контекста;
- hook-систему (`pre_tool`, `post_tool`, `stop`);
- OTEL tracing;
- durable resume через SQLite;
- опционально интеграцию с Inngest или Temporal.

### Обязательный артефакт

Подготовить post-mortem на ~1000 слов:

- что взято из Claude Agent SDK;
- что заимствовано из Deep Agents;
- что сознательно упрощено;
- что пришлось перепроектировать;
- какие trade-offs оказались ключевыми.

### Критерий завершения

Исполнитель понимает архитектуру обвязки как набор независимых слоев, а не как магию фреймворка.

## 8.5. Фаза 4. Evals и regression harness

### Цель

Сделать качество и стабильность агента измеримыми.

### Что нужно внедрить

1. Одну observability-платформу:

- LangSmith;
- Braintrust;
- Arize Phoenix / AX;
- W&B Weave;
- Inspect.

1. Четыре класса оценок:

- single-turn evals;
- trajectory evals;
- LLM-as-judge;
- end-state evals.

### Практический проект

Обернуть Phase 2-агента регрессионной eval-системой:

- собрать golden dataset на 30–50 вопросов;
- ввести уровни сложности;
- реализовать deterministic graders, где это возможно;
- добавить LLM judge по рубрике из 5 критериев;
- оценивать траекторию агента;
- встроить запуск evals в GitHub Actions;
- блокировать merge при регрессии;
- добавить nightly auto-grade для части production traces;
- протестировать хотя бы на одном внешнем benchmark;
- сделать `make eval` с итоговыми артефактами.

### Критерий завершения

Исполнитель умеет доказуемо фиксировать регрессии и контролировать качество изменений.

## 8.6. Фаза 5. Production hardening

### Цель

Подготовить систему к долгой эксплуатации в реальных условиях.

### Основные направления

#### 1. Стоимость

- prompt caching;
- model routing по сложности;
- контроль cost-per-task;
- batch API для не realtime-нагрузки;
- осторожное использование multi-agent сценариев.

#### 2. Latency

- parallel tool calls;
- streaming в UI;
- sub-agent fan-out там, где он оправдан.

#### 3. Безопасность

- sandbox для code execution;
- credentials вне контекста модели;
- guardrails через hooks;
- HITL-подтверждения на необратимые действия.

#### 4. Мониторинг и drift

- trace sampling;
- алерты по стоимости, latency, failure rate и quality;
- re-baseline evals после апгрейдов моделей;
- контроль устаревания assumptions в harness.

#### 5. Resilience

- durable execution для всех долгих сценариев;
- checkpoint на каждом узле;
- возможность resume / rewind / fork;
- тестирование путей восстановления заранее.

### Критерий завершения

Исполнитель способен сопровождать production-агента, а не только собирать демо.

## 10. Критерии приемки

Обучение считается выполненным, если по итогам всех фаз есть:

- минимум 1 рабочий агент без фреймворка;
- минимум 1 production-style агент на LangGraph;
- собственная мини-обвязка;
- зафиксированная система evals;
- CI-gates против регрессий;
- persistence и durable resume;
- документированные режимы сбоев;
- tracing и monitoring;
- README / demo-команды для воспроизведения.

## 11. Рекомендуемый стек

### Основной стек

- LangGraph 1.0 + Deep Agents.
- Claude Agent SDK как reference harness.
- LangSmith или Inspect для наблюдаемости и оценок.
- Tavily / Firecrawl для web research.
- PostgresSaver для persistence.
- Modal / E2B / Daytona для sandboxing.

### Использовать точечно

- OpenAI Agents SDK — если проект строго привязан к OpenAI.
- CrewAI — для быстрых прототипов и демо.
- Pydantic AI — для команд с сильным уклоном в typed FastAPI.
- Mastra — если стек команды полностью на TypeScript.
- Letta / MemGPT — если нужна self-managed память между сессиями.

## 12. Что не делать по умолчанию

- Не изучать 10+ фреймворков одновременно.
- Не начинать с multi-agent архитектуры без причин.
- Не внедрять vector DB до замера проблемы recall.
- Не считать демо-агента production-ready.
- Не внедрять evals «когда-нибудь потом».
- Не запускать опасный code execution без sandbox.

## 13. Основные риски

- Туризм по фреймворкам вместо инженерной глубины.
- Переоценка мульти-агентных систем.
- Отсутствие системы измерения качества.
- Рост стоимости при расширении контекста и числа суб-агентов.
- Деградация eval-suite со временем.
- Изменение поведения моделей после апдейтов.
- Шероховатость MCP в production-сценариях.

## 15. Финальная компетенция

После выполнения данного плана исполнитель должен быть способен:

- брать задачу на AI-agent feature end-to-end;
- спроектировать контекст, цикл и инструментальный слой;
- собрать агентную систему с устойчивой архитектурой;
- измерять качество и предотвращать регрессии;
- запускать и поддерживать production-решение.

## 16. Итоговая формулировка ТЗ

Необходимо пройти 6 фаз подготовки AI Engineer, завершив каждую фазу измеримым практическим артефактом. Основной результат программы — инженер, способный проектировать, собирать, оценивать и сопровождать production-ready agentic AI системы в 2026 году.
